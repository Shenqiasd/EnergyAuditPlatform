### C-3: 能源流向图 H2 表结构与 MySQL 生产 Schema 完全不一致

**H2 (schema-h2.sql)**: 单表 `de_energy_flow` — 扁平结构，每行一条 source→target 流向

**MySQL (sql/00-schema.sql)**: 三表范式设计：
- `de_energy_flow_diagram` — 主图表（含 `diagram_type`, `diagram_data` JSON）
- `de_energy_flow_node` — 节点（含 `node_id`, `node_type`, `ref_type`, `ref_id`, `position_x/y`）
- `de_energy_flow_edge` — 连线（含 `edge_id`, `source_node_id`, `target_node_id`, `energy_id`, `physical_amount`）

**差异分析**:

| 维度 | H2 `de_energy_flow` | MySQL 三表 |
|------|---------------------|------------|
| 结构 | 单表扁平 | 主表+节点+连线 |
| 节点坐标 | 无 | `position_x`, `position_y` |
| 图类型 | 无 | `diagram_type` (1/2/3) |
| 关联引用 | 无 | `ref_type`, `ref_id` → bs_* |
| X6 布局 | 无 | `diagram_data` JSON |

**影响**: 当前 Entity (`DeEnergyFlow`)、Mapper XML、Service、Controller 全部基于 H2 的扁平表设计。切换到 MySQL 生产环境时将完全无法工作。需要重新设计为匹配三表结构的 Entity/Mapper/Service。

### C-5: `submit` 端点缺少企业归属校验（IDOR 漏洞）

**文件**: `audit-web/src/main/java/com/energy/audit/web/controller/template/TemplateController.java:259-265`

```java
@PostMapping("/submission/{submissionId}/submit")
public R<Void> submit(@PathVariable Long submissionId,
                      @RequestParam Long templateVersionId) {
    requireEnterprise();
    // ⚠️ 未从安全上下文提取 enterpriseId 进行归属校验
    submissionService.submit(submissionId, templateVersionId);
    return R.ok();
}
```

对比 `saveDraft` 端点正确地提取了 `SecurityUtils.getRequiredCurrentEnterpriseId()` 并传递给 service。任何企业用户只要知道另一企业的 `submissionId` 即可提交其数据。

> 注: master 分支的 `SubmissionServiceImpl.submit()` 已内置了 enterpriseId 校验，但控制器层仍应传入以遵循防御性编程。
> ✅ 与 Devin Review Comment #6 一致，确认有效。

## 🟡 Medium 问题（合并前应修复）

### M-1: `EnergyFlowController` 缺少角色守卫

**文件**: `audit-web/src/main/java/com/energy/audit/web/controller/data/EnergyFlowController.java`

控制器的三个端点 (`GET /list`, `POST /save`, `DELETE /{id}`) 均通过 `SecurityUtils.getRequiredCurrentEnterpriseId()` 获取企业 ID（隐含要求企业用户），但没有显式的角色检查 (`requireEnterprise()`)。如果管理员或审核员的 token 中也有 `enterpriseId` 字段，则可能越权操作企业数据。

**修复**: 在每个端点开头添加 `requireEnterprise()` 调用，与 `TemplateController` 的模式保持一致。

---

### M-2: `releaseLock` 无事务保护，存在 TOCTOU 竞态条件

**文件**: `audit-service/src/main/java/com/energy/audit/service/template/EditLockService.java:85-98`

`releaseLock()` 先 `selectByKey`（无 FOR UPDATE）再 `deleteByKey`，两步之间可能被其他用户的 `acquireLock`（带 @Transactional + FOR UPDATE）穿插，导致新用户的锁被错误删除。

**修复建议**: 在 `deleteByKey` SQL 中增加 `AND lock_user_id = #{lockUserId}` 条件。

> ✅ 与 Devin Review Comment #4 一致，确认有效。

---

### M-3: `EntEnterpriseMapper.xml` updateById 无法将 `expireDate` 清空为 NULL

**文件**: `audit-dao/src/main/resources/mapper/enterprise/EntEnterpriseMapper.xml:74`

```xml
<if test="expireDate != null">expire_date = #{expireDate},</if>
```

`EnterpriseController.updateExpireDate()` 设计可接受 null 以清除过期日期，但 MyBatis 的 null-check 条件会跳过该列更新。

**修复**: 新增专用 `updateExpireDate` SQL 语句，无条件设置 `expire_date = #{expireDate}`。

> ✅ 与 Devin Review Comment #2 一致，确认有效。

---

### M-4: `SysRoleMapper.xml` updateById 缺少 `update_time` 自动更新

**文件**: `audit-dao/src/main/resources/mapper/system/SysRoleMapper.xml:43-54`

其他所有 Mapper XML 的 `updateById` 均包含 `update_time = CURRENT_TIMESTAMP`，唯独 SysRoleMapper 遗漏。角色更新后审计字段不会刷新。

> ✅ 与 Devin Review Comment #5 一致，确认有效。

### N-3: `EnergyFlowServiceImpl.saveBatch` 逐条插入性能问题

**文件**: `audit-service/src/main/java/com/energy/audit/service/data/impl/EnergyFlowServiceImpl.java:31-39`

```java
for (int i = 0; i < flows.size(); i++) {
    energyFlowMapper.insert(flow);  // N 次 SQL INSERT
}
```

当流向数据量较大时，逐条 INSERT 效率低。建议使用 MyBatis 的 `<foreach>` 批量插入优化。

### N-4: H2 开发 Schema 严重缺失（45 张表缺失）

**H2 schema-h2.sql** 仅包含 11 张表，而 MySQL 生产 schema 包含 56 张表。缺失的 45 张表涵盖：

- 系统模块: `sys_user_role`, `sys_role_menu`, `sys_menu`, `sys_dict_type`, `sys_dict_data`, `sys_config`, `sys_operation_log`
- 企业模块: `ent_registration`, `ent_enterprise_setting`
- 模板模块: `tpl_tag_mapping`, `tpl_submission`
- 数据录入模块: 全部 `de_*` 表（除 `de_energy_flow`）
- 审计报告/审核/碳排放/图表模块: 全部缺失

这意味着使用 H2 开发 profile 时，大部分功能无法测试。虽然这是渐进开发的正常状态，但应在文档中注明。
