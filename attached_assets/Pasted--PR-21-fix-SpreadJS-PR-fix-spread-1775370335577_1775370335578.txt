## 二、核心改动审核（SpreadJS License 配置）

### 2.1 `audit-ui/src/utils/spreadjs-license.ts`（新文件） ✅ PASS

```typescript
export function initSpreadJSLicense(): void {
  if (typeof window.GC === 'undefined') return

  const sheetsLicense = import.meta.env.VITE_SPREADJS_LICENSE
  if (sheetsLicense && window.GC.Spread?.Sheets) {
    window.GC.Spread.Sheets.LicenseKey = sheetsLicense
  }

  const designerLicense = import.meta.env.VITE_SPREADJS_DESIGNER_LICENSE
  if (designerLicense && window.GC.Spread?.Sheets?.Designer) {
    window.GC.Spread.Sheets.Designer.LicenseKey = designerLicense
  }
}
```

**评价**：
- ✅ 双 License Key 分离（Sheets + Designer），符合 SpreadJS V18 授权模型
- ✅ `typeof window.GC === 'undefined'` 安全守卫，CDN 未加载时优雅降级
- ✅ 可选链 `?.` 防止命名空间不完整时崩溃
- ✅ 环境变量通过 `import.meta.env` 读取，构建时注入，不暴露到运行时
- ⚠️ 相比 master 分支版本，**移除了 `initialized` 幂等保护**和 `console.warn` 提示。虽然多次调用 `initSpreadJSLicense()` 只是重复赋值同一个 key（无副作用），但丢失了 trial mode 的开发者提示

# 六、冲突处理方法论文档

## EnergyAuditPlatform 合并冲突处理指南

### 第一步：识别冲突类型

执行试探性合并查看冲突列表：

```bash
git merge --no-commit --no-ff origin/master
# 查看输出中的 CONFLICT 行
git merge --abort  # 不要真正合并，先分析
```

冲突分为三种类型，每种有不同的处理策略：

| 类型 | Git 标识 | 含义 | 典型场景 |
|------|----------|------|----------|
| **add/add** | `CONFLICT (add/add)` | 双方各自新建了同名文件 | 两个分支独立开发了同一个 Mapper/Entity |
| **content** | `CONFLICT (content)` | 双方修改了同一文件的重叠区域 | Dockerfile 中一方改了 ENV，另一方改了 RUN |
| **rename/delete** | `CONFLICT (rename/delete)` | 一方重命名，另一方删除 | 文件路径重组 |

### 第二步：分析冲突内容

对每个冲突文件，理解双方的改动意图：

```bash
# 查看冲突文件内容（包含 <<<<<<< / ======= / >>>>>>> 标记）
cat <冲突文件>

# 或使用 diff3 格式查看三方对比（HEAD / base / master）
git config merge.conflictstyle diff3
```

冲突标记格式：
```
<<<<<<< HEAD          ← PR 分支（当前分支）的版本
PR 分支代码
||||||| <base-hash>   ← 共同祖先的版本（diff3 模式下显示）
共同祖先代码
=======
master 分支代码
>>>>>>> origin/master ← master（目标分支）的版本
```

### 第三步：选择解决策略

#### 策略 A：保留一方（适用于 add/add 且一方是另一方的超集）

**判断标准**：如果一方包含另一方的全部功能并有额外改进，直接取更完整的版本。

**案例（PR #16）**：`ExtractedDataController.java`
- PR 分支：使用 `resolveEnterpriseId()` 支持三角色访问（审核员、企业、管理员）
- master：使用 `requireEnterprise()` 只支持企业角色
- **决策**：PR 分支是 master 的功能超集 → 保留 PR 分支（HEAD）

```bash
# 保留 PR 分支版本
git checkout --ours <file>
git add <file>

# 或保留 master 版本
git checkout --theirs <file>
git add <file>
```

#### 策略 B：合并双方改动（适用于 content 冲突且改动不重叠）

**判断标准**：双方修改了文件的不同部分，且改动互不冲突。

**案例（PR #18）**：`audit-ui/Dockerfile`
- PR 分支：新增 `ENV VITE_SPREADJS_LICENSE=$VITE_SPREADJS_LICENSE`
- master：将 `npm run build` 改为 `npx vite build`
- **决策**：两个改动独立 → 手动合并保留两者

```bash
# 手动编辑文件，删除冲突标记，保留双方改动
vim <file>
# 删除 <<<<<<< HEAD / ||||||| / ======= / >>>>>>> 行
# 保留双方有效代码
git add <file>
```

#### 策略 C：语义合并（适用于双方修改了同一逻辑）

**判断标准**：双方对同一段代码做了不同修改，需要理解业务意图后决策。

**案例（PR #21）**：`audit-ui/src/utils/spreadjs-license.ts`
- PR 分支：双 License Key（Sheets + Designer），无幂等保护
- master：单 License Key，有 `initialized` 幂等保护 + `console.warn`
- **决策**：需要手动合并 — 保留 PR 分支的双 Key 逻辑，考虑恢复 master 的幂等保护

```typescript
// 合并后的版本（示例）
let initialized = false

export function initSpreadJSLicense(): void {
  if (initialized) return
  if (typeof window.GC === 'undefined') return

  const sheetsLicense = import.meta.env.VITE_SPREADJS_LICENSE
  if (!sheetsLicense) {
    console.warn('[SpreadJS] VITE_SPREADJS_LICENSE not set — running in trial mode')
    return
  }
  if (window.GC.Spread?.Sheets) {
    window.GC.Spread.Sheets.LicenseKey = sheetsLicense
  }

  const designerLicense = import.meta.env.VITE_SPREADJS_DESIGNER_LICENSE
  if (designerLicense && window.GC.Spread?.Sheets?.Designer) {
    window.GC.Spread.Sheets.Designer.LicenseKey = designerLicense
  }

  initialized = true
}
```

#### 策略 D：rename/delete 处理

**判断标准**：一方重命名了文件，另一方删除了文件。

```bash
# 如果重命名后的文件是需要的
git add <新路径文件>
git rm <旧路径文件>  # 如果还存在

# 如果删除是正确的
git rm <文件>
```

### 第四步：批量处理 add/add 冲突

当大量 add/add 冲突文件属于同一模式（如本 PR 的 30 个后端文件），可以批量处理：

```bash
# 1. 列出所有 add/add 冲突文件
git status | grep "both added"

# 2. 对每个文件，用 diff 比较双方版本
git diff :2:<file> :3:<file>
# :2: = ours (PR branch), :3: = theirs (master)

# 3. 如果 PR 分支包含 master 的所有方法 + 额外方法，批量取 PR 版本
git checkout --ours <file1> <file2> <file3> ...
git add <file1> <file2> <file3> ...
```

### 第五步：验证

解决所有冲突后，务必验证：

```bash
# 1. 确认没有遗留冲突标记
rg "<<<<<<< " --type java --type xml --type ts --type vue

# 2. 后端编译
mvn compile -pl audit-web

# 3. 前端编译
cd audit-ui && npm run build

# 4. 四层对齐检查
# MySQL (sql/00-schema.sql) ↔ H2 (schema-h2.sql) ↔ Entity ↔ Mapper XML
```

### 决策流程图

```
冲突文件
  │
  ├─ add/add（双方新建同名文件）
  │    ├─ 内容完全相同 → 取任意一方
  │    ├─ 一方是超集 → 取超集方（策略 A）
  │    └─ 双方各有独特改动 → 语义合并（策略 C）
  │
  ├─ content（内容冲突）
  │    ├─ 改动在不同区域 → 合并双方（策略 B）
  │    └─ 改动在同一段代码 → 语义合并（策略 C）
  │
  └─ rename/delete
       ├─ 重命名有意义 → 保留重命名后的文件（策略 D）
       └─ 删除有意义 → 删除文件
```
